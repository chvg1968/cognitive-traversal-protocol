Cognitive Traversal Protocol

A community protocol for context-first LLM intervention

An LLM should not jump straight into intervention. First it reduces uncertainty, preserves invariants, declares impact, and only then changes the artifact. CTP turns that discipline into an executable routine: read less, understand better, change minimally, and verify in context.

Golden rule

If you cannot name what changes, what depends on it, and what remains unknown, you are not ready to intervene.

01 / Orientation

Mandatory start

The LLM's first job is to set scope. It is not solving yet: it is deciding how much context is required to act without breaking things.

1

Read the highest-authority instructions

Prioritize system instructions, environment rules, the user's brief, local documentation, and comments close to the artifact. If there is a conflict, record the contradiction and follow the applicable authority order.

2

Use external references only as a map

Wikis, docs, tickets, notes, and conversations help decide where to look. They never prove what exists today. Every datum, rule, contract, route, function, metric, or flow must be verified in the current source before being reported as fact.

First formal block 
TRIAGE:
  context_size: small | medium | large
  complexity: low | medium | high
  stack: [languages, frameworks, tools, or artifact types]
  core_purpose: "one verifiable sentence"
  task_scope: narrow | broad | full-audit
  external_refs_consulted: yes | no
  external_refs_sampled: [sources] | []
  external_refs_confidence: low | medium | high
  source_verification_required: yes | no
  required_sections: ["identity", "domain", "quality"]
  estimated_sources_to_read: number
  rationale: "why this scope is enough"

02 / Depth

Depth decision

CTP does not reward reading everything. It rewards reading the minimum needed to protect real contracts.

Situation Read first Do not
Small visual or narrative change Affected artifact, style guide, channel constraints Audit unrelated areas
Error with direct evidence Error message, indicated file or log, minimal reproduction Infer cause from names or assumptions
Contract, promise, or interface change Current definition, consumers, examples, tests, or validators Edit only the provider of the change
Data, identity, permissions, or compliance Data model, access rules, authority sources, critical history Continue with critical open questions
Cross-cutting restructure Inputs, outputs, shared modules, invariants, quality criteria Mix cleanup with hidden behavior changes
Useful heuristic: if a conclusion appears only in a secondary source, it is a hint. If it appears in the current artifact, its tests, data, configuration, or authoritative documentation, it can enter the anchor. If it appears nowhere verifiable, it remains an open question.

03 / Memory

Anchors before advance

An anchor prevents the LLM from reasoning from scratch again and reduces drift between sections.

Observe

Read with limits

Sample representative artifacts, entry points, contracts, and real examples. Record what was read and what was intentionally skipped.

Compress

Compress into state

Turn reading into invariants, dependencies, risks, contradictions, and open questions.

Consolidate

Look for conflict

Compare against previous anchors. A critical conflict stops the work until it is resolved.

Advance

Advance with memory

The next section uses the anchor, not a new intuition or a vague memory.

Reading budget: cap reading at the smaller of 15 sources or 3,000 lines per section. Sample representative artifacts — entry points, contracts, schemas, interfaces — and record what was sampled vs. skipped intentionally. If the section is larger, note what was skipped and why. 
ANCHOR[section-name]:
  summary: "2 to 4 lines about what was observed"

  invariants:
    - text: "property that must be preserved"
      priority: critical | important | soft

  dependencies:
    - "source -> consumer"

  risks:
    - text: "specific risk"
      priority: critical | important | soft
      severity: low | med | high

  contradictions:
    - against: "ANCHOR[previous-section]"
      text: "previous finding vs current finding"
      priority: critical | important | soft
    # or: none

  open_questions:
    - "question that still matters"
    # or: none

  sources_sampled: ["verified sources"]
  sources_skipped_intentionally: ["sources or patterns"]

04 / Impact

Map before intervening

The IMPACT_MAP is the traffic light. If it contains unknowns, read more, ask, or stop.

Blocking 
IMPACT_MAP:
  change_summary: "what will change"
  direct:
    - "artifact / module / contract directly edited"
  indirect:
    - "behavior affected through dependency"
  unknown:
    - "suspected but unconfirmed area"
  affected_invariants:
    - text: "affected invariant"
      priority: critical | important | soft
      from_anchor: "ANCHOR[section-name]"
  minimal_modification_justification: "why this change is enough"
Direct

What you will intervene on

Files, documents, datasets, contracts, components, functions, or processes that will receive the change.

Indirect

What can change by dependency

Consumers, flows, styles, metrics, routes, tests, or decisions touched by the effect.

Unknown

What blocks

Any relevant area you cannot classify. Unknown is not a note: it is a stop.

05 / Intervention

Intervene without drift

The correct modification is the smallest one that satisfies the task and respects the consolidated anchors.

Preserve contracts

Do not change signatures, schemas, routes, permissions, definitions, metrics, or public semantics as a side effect. If you must, declare it as a direct change.

Respect the local pattern

Use existing patterns from the system, document, dataset, or team before inventing abstractions. Coherence is worth more than novelty.

Do not hide uncertainty

If an unexpected dependency appears during intervention, pause and update the map. Do not keep acting with blurry impact.

Separate refactor and behavior

When possible, avoid mixing structural cleanup with changes in meaning. If they mix, explain why it was necessary.

Operational close checklist

  1. The change matches the declared scope.
  2. No unknowns remain in the IMPACT_MAP.
  3. Critical invariants remain intact.
  4. Dependencies still point in the same direction.
  5. The solution adds no complexity without a clear benefit.

06 / Verification

Verifiable close

An LLM is done when it can state what it verified, what it could not verify, and what risks remain.

Change type Minimum verification Report if missing
Document or static page Structure inspection, render or full read-through, internal links No browser, viewer, or validator available
Application or automation Lint, typecheck, relevant tests, local run if applicable Dependencies not installed, blocked environment, or missing credentials
Contract, API, or integration Expected input/output, consumers, errors, compatibility External services unavailable
Data, models, or business rules Verifiable sample, rollback, compatibility with current source Data source inaccessible or insufficient sample
